**Date:** 2026-07-02

## Basis

1. **WS5 hardening campaign (rounds 290–306, 2026-07-01/02).** Full-denominator
   as-run log in [`M2.8.1-WS5-Campaign.md`](M2.8.1-WS5-Campaign.md):
   announcement-mismatch injection sealed as `announcement_mismatch` (round
   293), sidecar liveness fault injected and self-recovered without manual
   restart (295), restart-safety AC1.4 (296), and a 10-round clean streak
   (297–306) with three-level matches and zero divergence.
2. **Zero successful mid-window output fetches.** Across every tracked round of
   the campaign and the addendum, every external output probe during an open
   commit window returned 404 while the frozen input package stayed public, no
   `final_bundle_cid`, GitHub Pages URL, or final memo hash appeared on the
   rounds API mid-window, the public VL sequence never advanced mid-window, and
   the in-container echo red-team probe never obtained foundation output
   hashes.
3. **Post-campaign regression found, fixed, and verified live (addendum rounds
   307–310, 2026-07-02).** Independent review of the hardening found that VL
   activation was still anchored to signing time, so deferred publication
   distributed already-effective lists (staggered activation — the fork-risk
   window the M1.10.6 lookahead closes). Fixed in `100f2af`
   (publication-anchored `effective` plus a delayed-resume re-sign guard) and
   proven on devnet:
   - **307** — published VL activation 18:06:27 = publication due + lookahead,
     in the future at distribution (pre-fix round 306 was ~6 min past); plus a
     full sidecar recovery from operator-stopped Modal apps, sealing 3/3 valid.
   - **308** — restart mid-hold: resumed and published exactly once with the
     still-fresh persisted stamp (guard correctly silent); 3/3 valid.
   - **309** — outage held past stamp staleness: the guard re-signed
     (`was 18:53:09, now 19:09:38`), published exactly once, and the audit
     bundle and distributed VL hashed identically; the three `missing_reveal`
     outcomes are the injected outage covering the reveal window, classified
     per protocol with no VL disruption.
   - **310** — clean hands-off round: withholding probes clean, fresh
     publication-anchored stamp, 3/3 valid at all three levels, convergence
     sealed and anchored.
4. **Sidecar liveness follow-ups** (`7c191e0`): operator-configurable inference
   read timeout with commit-deadline cap precedence, a watchdog budget guard,
   and a default verify-only watchdog — test-covered in the sidecar repository
   and running on all three devnet participation instances throughout the
   addendum rounds.

## Known observations (tracked follow-ups, not blockers)

- The rounds API does not expose `output_publication_commit_closes_at` /
  `output_publication_due_at` (the database carries them). The sidecar's
  deadline capping falls back to its own recorded announcement windows, which
  works; exposing the fields would make the API contract match the sidecar's
  metadata reader.
- Devnet's 0.12 h activation lookahead is operationally tight against the
  scheduler tick plus GitHub Pages CDN propagation: the stamp is correct at
  distribution, but validators can fetch after activation once CDN delay is
  added. Testnet's 0.5 h absorbs this comfortably. Consider raising devnet to
  ~0.25 h.
- A sidecar whose recorded Modal app is stopped out-of-band reports
  `INFERENCE_ERROR {"status_code": 404}` until a one-shot `deploy-modal`
  rewrites the record (proven in round 307); zero-touch provisioning
  intentionally never replaces an existing record.
- A foundation-service outage that spans a reveal window costs sidecar reveals
  (`missing_reveal`), because each participate pass resolves round metadata
  from the foundation API (round 309). Chain-side submission itself does not
  depend on the service.

## Decision

M2.8.1 is complete. M2.9 (testnet shadow rollout) is unblocked: the hardened
foundation code and the activation-anchoring fix are live on devnet and
testnet, and the sidecar testnet images are published. M2.9 requires sidecar
deployments on the three foundation-operated testnet validators, one clean
weekly testnet verification round with an as-run record in `docs/phase2/`, and
an explicit owner decision for any public announcement.
