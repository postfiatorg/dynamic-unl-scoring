## Findings and Follow-ups

### 🔴 BUG — Scheduler path fails on on-chain memo

Round 5, initiated by the scheduler, failed on the `ONCHAIN_PUBLISHED` step:

```
PFTL memo transaction error: asyncio.run() cannot be called from a running event loop
Round 5 failed: ONCHAIN_PUBLISHED: On-chain memo submission failed
```

**Root cause:** `scoring_service/clients/pftl.py` imports `autofill` and `submit_and_wait` from `xrpl.transaction`. These are **async functions** in xrpl-py; when called from a synchronous context, xrpl-py internally calls `asyncio.run()` on them. The scheduler (`scheduler.py`) runs inside FastAPI's lifespan as an asyncio task, so when it hands off to the orchestrator and the orchestrator eventually calls the PFTL client, `asyncio.run()` blows up because an event loop is already running in the current thread.

The manual trigger path does not hit this because `scoring_service/api/scoring.py` explicitly launches `_run_round_in_background` inside a `threading.Thread`, which has no event loop. Round 6 (manual trigger) therefore completed normally while Round 5 (scheduler) failed.

**Impact:** The weekly automated scoring cadence will fail every time the scheduler fires. This must be fixed before M1.10.7 iteration runs or before the scheduler next wakes up.

**Suggested fix:** Import from `xrpl.asyncio.transaction` and await the coroutines, OR refactor `PFTLClient.submit_memo` to explicitly offload work to a thread via `asyncio.to_thread()` when called from an event loop context. The cleanest fix is probably to use the sync helpers directly — but they may not exist in xrpl-py for all operations; worth checking the xrpl-py source.

### ⚠️ GAP — VHS returns 0 devnet topology nodes

`GET https://vhs.devnet.postfiat.org/v1/network/topology/nodes` returns `{"result":"success","count":0,"nodes":[]}` for devnet. Because the collector iterates over topology nodes to probe `/crawl` and build the IP → validator mapping, it ended up with zero IPs, which cascaded into:

- 0/6 validators with IP resolution
- 0/6 validators with ASN enrichment
- 0/6 validators with country-level geolocation
- LLM scored all validators with `diversity=30` (full diversity penalty) and flagged the concentration risk in the network summary

**Impact:** The scoring was essentially blind to geographic diversity for this round. The two temporary validators (Frankfurt, Singapore) brought real ASN and country diversity but received no credit for it because the enrichment data was missing.

**Suggested investigation:**

1. Check VHS connection manager logs on the devnet VHS host to confirm whether the crawler has rediscovered nodes since the v1.0.3 genesis reset
2. If the crawler needs more time after the reset, re-run a scoring round once topology accumulates
3. If there's a deeper VHS issue (e.g., crawler bound to an old peer IP), open an issue in `validator-history-service`

### 📝 NOTE — Pinata gateway 404 immediately after pinning

The public Pinata gateway (`https://gateway.pinata.cloud/ipfs/<cid>/...`) returns 404 for the newly-pinned CID even 30+ seconds after the pin. This is **expected behavior** — Pinata's `pin-by-CID` endpoint is asynchronous. It accepted the request (HTTP 200) and the pin is fully registered in Pinata's internal pin list (verified directly via `GET /data/pinList`), but the content has not yet been fetched from the IPFS network and cached by the gateway's edge.

**Evidence the pin is real:**

```json
{
  "id": "7a423aca-c798-428e-9f82-a896053a0718",
  "ipfs_pin_hash": "Qmf5jUdvTN3TJRHG75qkuUCKeCJ8fK1PshtikQGy5Ue6MM",
  "date_pinned": "2026-04-10T13:37:11.363Z",
  "regions": [{"regionId":"pinata","currentReplicationCount":1,"desiredReplicationCount":1}]
}
```

**Why it matters (slightly):** Pinata needs to fetch the CID from the public IPFS DHT. The primary IPFS node (`ipfs-testnet.postfiat.org`) may not be fully reachable from Pinata's workers because it's behind basic auth. To accelerate propagation in future rounds, consider passing the primary node's multiaddress via Pinata's `hostNodes` field in `pin_by_cid()`:

```python
"pinataOptions": {"hostNodes": ["/dnsaddr/ipfs-testnet.postfiat.org/.../<peer_id>"]}
```

This is a cosmetic improvement — the scoring pipeline and the HTTPS fallback already serve the audit trail correctly, and Pinata's internal pin record is the only thing that matters for the redundancy guarantee.

### 📝 NOTE — Tied top-4 scores, khorne lost the tiebreak

The four foundation validators scored identically (75 across all dimensions). The UNL selector picked slaanesh, nurgle, and tzeentch, leaving khorne as the first alternate. This is because the LLM's validator list is emitted in a consistent order per prompt but is arbitrary with respect to validator identity when scores are tied. For scoring stability testing (M1.10.8), identical scores should not produce unstable selections across runs — worth verifying that the tiebreak is deterministic on replay.
