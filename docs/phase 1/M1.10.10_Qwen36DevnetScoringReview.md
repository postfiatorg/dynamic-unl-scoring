## Scope

## Prompt Quality Findings

The current v2 prompt succeeds at the most important behaviors:
- It produces valid, complete JSON.
- It differentiates validators rather than clustering everything into one score.
- It recognizes the network-level Vultr concentration risk.
- It recognizes country distribution across United States, Germany, and Singapore.
- It penalizes missing domain accountability.
- It treats all validators as current on software version `1.0.4`.
- It explains the selected scores with evidence from the snapshot.

The review also found three prompt weaknesses worth fixing before testnet.

First, v2 gives current UNL membership too much space inside the LLM reliability dimension. The selector already applies explicit churn control after scoring. Having the model also reward incumbency risks double-counting the same continuity preference. In devnet this showed up when V4 had verified domain, excellent agreement, current software, and the same diversity profile as V1/V2/V3, but remained scored at `82-83` while not on the UNL. The final selector behaved correctly, but the prompt should make clear that UNL membership is context, not a substitute for current evidence.

Second, the identity sub-score was too loose. The underlying evidence did not change meaningfully, but verified-domain validators moved between identity scores of `50`, `70`, and `80` across rounds. Validators with no domain moved between `30` and `50`. The overall scores stayed stable, but the dimensional scores are public-facing audit data and should be more interpretable. The prompt needs clearer identity anchors: null formal identity is neutral when identity verification is not deployed, verified domain is a positive accountability signal, and no domain remains a penalty.

Third, the software wording relied too much on "most common version" as a proxy for current software. That can be misleading on testnet during upgrades, where the majority may temporarily be on an older release while better-maintained validators upgrade first. The prompt should reason from version ordering and visible release evidence rather than penalizing a newer version because it is uncommon.

The diversity behavior was acceptable in these rounds, but the prompt benefits from tighter framing. V5 and V6 are geographically useful because Germany and Singapore are underrepresented, but they are not true infrastructure diversification because they share the same Vultr ASN as every other devnet validator. The v3 prompt makes that country-vs-ASN distinction explicit.

## Decision

Create `prompts/scoring_v3.txt` and make it the active prompt.

This is a versioned prompt revision rather than an in-place edit to `scoring_v2.txt`. Historical v2 rounds remain auditable as v2. Future rounds should publish `prompt_version = v3` in `scoring_config.json`.

The v3 prompt keeps the v2 output contract intact:
- same top-level JSON shape
- same validator ID behavior
- same `network_summary`
- same dimensional sub-score fields
- same 0-100 score ranges

The v3 changes are intentionally narrow:
- Current UNL membership is context for continuity, not proof of intrinsic validator quality.
- Identity scoring is anchored around verified domain vs no domain while treating null formal identity as neutral.
- Software scoring avoids treating a newer minority version as bad merely because it is uncommon.
- Diversity scoring explicitly separates country diversity from ASN/provider diversity.
- Per-validator reasoning asks for concrete evidence and the main limiting factor while keeping the output concise.
