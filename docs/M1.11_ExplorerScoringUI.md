# Explorer Scoring UI — Design Spec

Design spec for the scoring surfaces added to the block explorer in Milestone 1.12.

**Naming note:** the filename reads `M1.11_` for historical reasons — the scope was originally tagged M1.11 and renumbered to M1.12 when admin override endpoints became M1.11. The filename is kept to avoid breaking links from commits, PRs, and the roadmap; the milestone is M1.12.

**Three surfaces are touched:**

1. **Validators page** (existing) — the binary UNL column is replaced by a single combined badge column carrying both the numeric score and the UNL status.
2. **UNL Scoring page** (new) — a dedicated page in the explorer navigation.
3. **Validator detail page** (existing) — gets a compact scoring section with status, overall score, and dimension breakdown.

The spec is opinionated about information architecture, data sources, routing, error handling, and component reuse. Typography, spacing, and exact color values are deferred to implementation but must reuse existing explorer primitives where they exist (`MetricCard`, `StatusBadge`, `CopyableAddress`, the `getAgreementColor` ramp, `dashboard-panel` layout).

## Page 1: Validators Page (Enhanced)

The existing validators table (`explorer/src/containers/Network/ValidatorsTable.tsx`) currently has columns: **Pubkey**, **Domain**, **UNL** (a binary green checkmark when the validator is on the UNL), **Version**, agreement scores **1H / 24H / 30D** (on the uptime tab), fee-voting fields (on the voting tab), and **Last Ledger**.

The existing binary **UNL** column is **replaced by a single combined badge column** — `Status`. The badge carries both the numeric overall score and the UNL status in one visual unit so they always travel together and sort together.

Before (relevant slice):

| Pubkey | Domain | UNL | Version | Agreement 1H | 24H | 30D | ... |
|---|---|---|---|---|---|---|---|
| nHUKG1ZY… | postfiat.org | ✓ | 1.0.3 | 99.8% | 99.7% | 99.8% | ... |

After (relevant slice):

| Pubkey | Domain | Status | Version | Agreement 1H | 24H | 30D | ... |
|---|---|---|---|---|---|---|---|
| nHUKG1ZY… | postfiat.org | `● 82 on UNL` | 1.0.3 | 99.8% | 99.7% | 99.8% | ... |

The three Agreement columns (1H / 24H / 30D) are **unchanged** — preserved in their current form. The Version, Fee Voting, and Last Ledger columns are also unchanged.

**The table becomes sortable.** At minimum, `Status` (by score descending, default), `Agreement 30D`, `Version`, and `Last Ledger` must be clickable column headers with visible sort arrows. Implementing sort-on-click is a required part of this milestone, not a nice-to-have; the current table's lack of sorting is a gap this work closes.

### Status values

The four states are mutually exclusive and exhaustive for any validator visible on the page. Each uses a distinct glyph so status is never color-only (accessibility):

| Glyph | Label | Condition | Data source |
|---|---|---|---|
| ● | `on UNL` | `score ≥ cutoff` AND in the final selected UNL | `unl.json` → `primary` |
| ◐ | `candidate` | `score ≥ cutoff` AND NOT in the final UNL | `unl.json` → `alternates` |
| ○ | `ineligible` | `score < cutoff` | present in `scores.json`, absent from `unl.json` |
| — | `no data` | validator absent from this round's scoring evidence | absent from `scores.json` |

A validator is in exactly one category per round. A validator **cannot** be both `candidate` and `ineligible` — the selection algorithm only draws alternates from the pool above the cutoff.

**Why "no data" happens.** The explorer's Validators page lists every validator VHS has ever observed on the network (UNL + community). The scoring pipeline is stricter: it only scores validators whose enrichment data could be resolved (cached manifest, IP reachable via `/crawl`, ASN lookup, geolocation). A community validator with a flaky manifest or unresolvable IP appears in VHS but not in `scores.json`. Most validators will have data; a minority will not.

Note on churn: in rounds after the first, `alternates` can include validators that were previously on the UNL but got displaced by stronger challengers this round. They are still `candidate`s (above cutoff, not selected). The UI does not distinguish displaced incumbents from first-time challengers.

**Badge styling.** Green (on UNL), amber (candidate), grey (ineligible), light grey (no data). Colors are secondary signaling only — the glyph conveys state.

### Default sort

**Status column, descending by score.** `on UNL` rows sort first (by score desc), then `candidate`, then `ineligible`, then `no data`. Ties within a state break by agreement 30D desc.

### Freshness & staleness signaling

The footer line `Scores from round #N — completed X ago.` visually escalates with staleness, computed from `completed_at + cadence_hours` (cadence from `/api/scoring/config`):

- **Fresh** (< cadence + 24h): neutral text.
- **Overdue** (> cadence + 24h): amber text.
- **Severely overdue** (> 2× cadence): red text.

No popup, no modal — just a color cue so operators notice when rounds haven't been running.

### Navigation

- The badge is display-only.
- The `[ View scoring details → ]` link below the table goes to `/unl-scoring`.
- Clicking a validator row still goes to the validator detail page (unchanged).

## Page 2: UNL Scoring (New Page)

Added to the top-level explorer navigation between **Validators** and **Amendments**. Single-page layout with stacked sections.

### Section A — Header banner: round state

The banner's appearance depends on pipeline state. Three variants.

#### A.1 — Idle (last round complete, scheduler waiting)

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  UNL SCORING                                                                 │
│                                                                              │
│  Last round:   #14                 Next round:   in 5 days, 3 hours          │
│  Completed:    2 days ago          Cadence:      weekly (168h)               │
│  Status:       ● COMPLETE          Scheduled:    2026-04-19 14:22 UTC        │
│                                                                              │
│  Health:  ● Scheduler · ● LLM endpoint · ● Publisher wallet                  │
└──────────────────────────────────────────────────────────────────────────────┘
```

No LLM-generated "network summary" is shown. This is a verification-oriented page; the idle banner keeps to operational facts that are directly checkable.

**Health strip** (right side of banner, or full-width row below it): three dots for scheduler liveness, LLM endpoint reachability, and publisher wallet readiness. Each green / amber / red with a short tooltip explaining the signal. Exact data source: derived from the same health checks the scoring service already runs for `/v1/health`-style endpoints — no new backend work required beyond surfacing existing fields.

#### A.2 — In progress (a round is currently running)

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  UNL SCORING                                                                 │
│                                                                              │
│  Round #15 running — started 22 seconds ago                                  │
│  [ auto-refresh every 5s ]                                                   │
│                                                                              │
│  Health:  ● Scheduler · ● LLM endpoint · ● Publisher wallet                  │
└──────────────────────────────────────────────────────────────────────────────┘
```

Stage-by-stage pipeline progress is **not** shown. Rounds complete in a few minutes; a live stage timeline adds complexity and a backend field without matching operator value for v1. The banner only needs to communicate "a round is running, since when." Stage breakdown, if needed later, belongs in a separate operator-only debug view.

#### A.3 — Failed (most recent round failed)

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  UNL SCORING                                                                 │
│                                                                              │
│  Round #15  ✕ FAILED at stage ONCHAIN_PUBLISHED (2 hours ago)                │
│  Error:    "PFTL submit timeout after 30s — RPC node unreachable"  [ more ▼ ]│
│                                                                              │
│  Last successful round: #14 (2 days ago) — UNL below is from round #14       │
│  Next scheduled attempt: in 5 days, 3 hours                                  │
│                                                                              │
│  [ View round #15 details → ]                                                │
│                                                                              │
│  Health:  ● Scheduler · ◐ LLM endpoint · ● Publisher wallet                  │
└──────────────────────────────────────────────────────────────────────────────┘
```

- **Expandable error**: long error strings (multi-line stack traces, repeated attempt logs) collapse behind `[ more ▼ ]`; one-liners show inline.
- **Direct link** to the failed round's detail page (`/unl-scoring/rounds/15`). No "tap the error" ambiguity — one clearly-labeled link.

### Section B — Ranked validator table

The core view. All scored validators, sorted by overall score descending. Two separator lines correspond directly to the two configurable parameters of the selection algorithm:

1. **UNL selection line** — between the last validator selected onto the UNL and the first candidate.
2. **Cutoff line** — between the last candidate and the first validator below the eligibility cutoff.

Separator lines render as labelled chips floating on the divider:

```
— UNL cutoff · top 35 —
— eligibility cutoff · score ≥ 40 —
```

Values in the chips are rendered from `/api/scoring/config`, never hardcoded.

#### Table layout

#### Columns

| Column | Content |
|---|---|
| Rank | Position in score order |
| Validator | Truncated pubkey with `CopyableAddress`; clicking a row expands inline drill-down |
| Overall | Numeric score 0–100 |
| **Δ** | Change vs previous round: `↑3`, `↓1`, `=` (no change), `new` (first appearance), `displaced` (was on UNL last round, no longer) |
| Consensus | Bar (0–100) using `getAgreementColor` ramp |
| Reliability | Bar |
| Software | Bar |
| Diversity | Bar |
| Identity | Bar |

The 5 dimension columns use horizontal filled bars (matching the existing agreement-bar pattern) so readers see *shape* — weak on Identity, strong on Consensus — without parsing digits.

#### Filter / search

Top-right of the table: a text input filtering rows by pubkey substring or domain substring. Debounced, client-side. Filtered-out rows hide but the separator lines stay in place — so a validator op searching for themselves can see their row relative to the cutoffs.

#### Sticky headers

Table column headers stick to the top of the viewport when the user scrolls down the list. Cheap to add in v1; painful to retrofit once users rely on it.

#### Churn-gap visualization

The `min_score_gap` rule ("challenger must score at least X points higher than the weakest incumbent to displace") is currently invisible to readers. Surface it as a chip on the UNL-selection separator line (shown above: `— UNL cutoff · top 35 ─── weakest on UNL: 62 ──`) and a matching chip on the eligibility line (`min gap to displace: 5`). A candidate at score 63 that clears the cutoff but *doesn't* clear the gap gets a subtle amber outline so the reader sees "above cutoff, still won't get in this round."

#### Empty zones

Any of the three zones (on-UNL / candidate / ineligible) may be empty, especially on devnet (UNL size 3, small validator set). Empty-zone rendering:

- **Empty candidate zone** (UNL filled and no one above cutoff waiting): show a single-row placeholder `— No candidates this round —` between the two separator lines.
- **Empty ineligible zone**: placeholder `— No ineligible validators this round —` below the cutoff line.
- **Both empty**: merge both lines into one labeled `— All scored validators are on the UNL —` to avoid stacking two adjacent horizontal rules with nothing between them.

**Important differences from earlier draft:**

- **No IP address.** Publishing validator IPs on a public explorer page is a DDoS-targeting gift. ASN + country provide the operational signal (diversity verification) without naming individual hosts.
- **Score history sparkline**: tiny inline chart (~60px × 20px) showing this validator's overall score across the last ~10 rounds. Lets a reader see trend at a glance — rising, flat, or dropped.
- **Two separate download buttons** — one per artifact. "Download snapshot entry" returns just this validator's slice of `snapshot.json`; "Download score entry" returns just this validator's slice of `scores.json`.
- **Link to the validator detail page** — operators who want full context (transaction history, voting history, agreement charts) click through.
- **Reasoning stays as one block.** Per-dimension reasoning is not available from the upstream LLM in the current pipeline. If that changes later, structure the block by dimension.

#### Columns

| Column | Content |
|---|---|
| Round | Round number |
| Date | Relative ("2 days ago") with absolute on hover |
| Status | `● COMPLETE`, `✕ FAILED`, `● RUNNING`, `● DRY_RUN_COMPLETE` |
| **Trigger** | `scheduled`, `manual`, `override` — distinguishes automated from human-initiated rounds; override rounds (M1.11) show the `override_reason` in the audit-trail panel when selected |
| UNL size | Number of validators selected (empty for failed rounds) |
| **Failed at** | Stage name on failure (`COLLECTING`, `SCORED`, `SELECTED`, `VL_SIGNED`, `IPFS_PUBLISHED`, `VL_DISTRIBUTED`, `ONCHAIN_PUBLISHED`) — empty for successful rounds. Lets operators spot patterns ("last 3 failures at ONCHAIN_PUBLISHED → wallet issue") without drilling into each round |

#### Row selection → URL update

Clicking a row updates the URL to `/unl-scoring/rounds/:roundId` and re-renders sections A, B, C, E, F for that round. A breadcrumb at page top reads `Viewing round #13 · [Back to latest]` when historical. URL state is shareable — operators can paste a link into Slack that shows exactly the round they're discussing.

#### Pagination

Initial load: `?limit=20`. "Load more" button fetches `?offset=20`, `?offset=40`, etc. Not urgent today (~15 rounds total) but cheap to build now and painful to retrofit once hundreds of rounds exist.

The verification surface. Without this, the page is unverifiable.

```
│               [ Open on ipfs-testnet.postfiat.org ]                          │
│               [ Open on Pinata gateway ]                                     │

│  Published VL (this round):                                                  │

│    Download:        [ vl.json (this round) ]                                 │

│    [ View transaction on explorer ]                                          │
│                                                                              │
│  GitHub Pages commit:                                                        │
│    [ View commit on postfiatorg.github.io → ]                                │
│                                                                              │
│  Artifacts on IPFS:                                                          │
│    snapshot.json · scores.json · unl.json · vl.json · metadata.json          │

└──────────────────────────────────────────────────────────────────────────────┘
```

#### Key details

- **Gateway names are explicit.** "Primary gateway" is replaced with the literal hostname (`ipfs-{env}.postfiat.org`) so users understand one is PostFiat-hosted and one is Pinata.
- **No SHA-256 file hashes are displayed.** An IPFS CID *is* a content hash — any tampering with files on IPFS changes the CID, which then mismatches the on-chain memo. That verification chain is sufficient; per-file SHA-256s on the page would be redundant noise without a self-serve verifier (out of scope for v1).
- **VL sequence** and **VL expiration** are shown. Round number ≠ VL sequence (failed rounds advance round counter but not VL counter). Both matter for audit and operational monitoring (the VL needs to be re-published before it expires).
- **Per-round VL link.** The `vl.json` link resolves to `/api/scoring/rounds/<N>/vl.json`, not the always-latest `/vl.json`. Viewing round #13 must download round #13's VL, not round #14's.
- **Override rounds**: when `Trigger = override`, this panel adds a row `Override reason: "..."` showing the `override_reason` field from the round record.

#### Failed rounds

If the selected round is `FAILED`, the audit trail panel collapses to:

```
AUDIT TRAIL — Round #15

No audit trail — round did not publish.

See the Round history for the failure stage, and the Header banner for the error message.
```

### Section F — Methodology explainer

Collapsed accordions at the bottom of the page. Two items, not four.

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  ▸ How scoring works                                                         │
│  ▸ How to verify                                                             │
└──────────────────────────────────────────────────────────────────────────────┘
```

Expanded "How scoring works":

```
Validators are scored 0–100 by an open-weight LLM across 5 dimensions:
Consensus, Reliability, Software, Diversity, Identity.

A validator must score at or above the eligibility cutoff (currently {cutoff})
to qualify. The top {max_size} qualifying validators are placed on the UNL
for the coming round.

To prevent churn, a challenger must score at least {min_gap} points higher
than the weakest incumbent to displace it.

Scoring runs every {cadence_hours} hours. Each round produces auditable
artifacts (snapshot, scores, UNL, signed VL, metadata) pinned to IPFS and
anchored on-chain.
```

Expanded "How to verify":

```
Every round produces a content-addressed bundle on IPFS (shown in the Audit
Trail panel). The on-chain memo transaction includes the IPFS CID and VL
sequence, so the chain is:

  on-chain memo  →  IPFS CID  →  signed VL  →  validators load and use it

Since the IPFS CID is a content hash, any tampering with the published
artifacts changes the CID, which then mismatches the memo — making tampering
detectable by anyone who can read the memo and the IPFS bundle.
```

**Per-dimension definitions** (what Consensus vs Reliability etc. actually measure) are **not** a top-level accordion item. They live as tooltips on the dimension column headers in Section B — where a user is actually looking at dimension values.

Values in braces (`{cutoff}`, `{max_size}`, `{min_gap}`, `{cadence_hours}`) are fetched live from `/api/scoring/config`. They are never hardcoded. The `/api/scoring/config` endpoint is a **hard dependency** of this milestone, not an addendum — the banner countdown (Section A) and the methodology values both require it.

## Page 3: Validator Detail Page (Enhanced)

The existing validator detail page (`/validators/:identifier`) gains a compact **Scoring** section placed **between the existing agreement-bars overview grid and the tabs**. The panel uses the standard `dashboard-panel` chrome (border + background) shared with the hero card above the agreement row; no new panel type is introduced.

### Layout

Two-half body inside a single panel:

- **Left half** — fixed-width column containing a circular `ScoringRing` (SVG, default 120 px) rendering the overall score as a color-coded progress arc with the score number centered inside. Directly below the ring: the `ScoringStatusBadge` rendered with `hideScore` so the status label (`● on UNL`, `◐ candidate`, `○ ineligible`) is not redundant with the number already shown inside the ring.
- **Right half** — flex-growing column of five stacked rows, one per dimension, laid out as a three-column grid (`LABEL` · horizontal bar · numeric value). Label is fixed-width uppercase, bar is fluid width at 6 px height, value is right-aligned. Stacking the five dimensions vertically makes them comparable on a single eye-sweep down aligned baselines, which is more readable than a row of five cards.

The panel header is a single word (`SCORING`) in the same uppercase-label style used elsewhere on the detail page. The round number and relative time are deliberately omitted from this panel — the visualization already conveys "latest completed round" by its place on the page, and readers who want round-specific context click through via the footer link.

The ring and the five bars share one color ramp (green / yellow / orange / neutral). The overall ring uses **status-derived** colors (on-UNL → green, candidate → yellow, ineligible → orange, no-data → neutral) so the ring color is legible even if the viewer doesn't know the numeric score. The five dimension bars use **score-threshold** colors: green ≥ 70, yellow ≥ 40, orange below (40 matches `unl_score_cutoff` from `/api/scoring/config`, so "below orange" reads as "below the scoring eligibility bar").

The `[ View reasoning and round history → ]` link in the panel footer navigates to `/unl-scoring/rounds/<latest>?validator=<pubkey>`, which opens the Scoring page with this validator's drill-down auto-expanded. That route is produced in later milestone steps; the link is forward-looking and 404s until the Scoring page lands.

### Dimension tooltips

Each dimension row has `cursor: help` and a `title` tooltip carrying canonical copy defined in `SCORING_DIMENSIONS` (see `explorer/src/containers/Network/scoringUtils.ts`). The strings are intentionally short and rubric-aligned with the LLM prompt in `scoring_service/prompts/scoring_v2.txt`:

| Dimension | Tooltip |
|---|---|
| Consensus | How reliably the validator's proposals match consensus across recent ledgers. |
| Reliability | Operational reliability signaled by verified domain (public accountability) and current UNL membership. |
| Software | Whether the validator runs up-to-date software and votes reasonable fees. |
| Diversity | Geographic and infrastructure spread. Validators in underrepresented countries or on less common cloud providers score higher. |
| Identity | Identity and reputation. Verified domain and organizational identity raise this score. |

These tooltips are the single source of truth for dimension copy across every scoring surface in the explorer — the Scoring page (M1.12.7) must import the same `SCORING_DIMENSIONS` rather than redefining strings.

### No-data case — validator not in the latest round

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ SCORING                                                                      │
│                                                                              │
│ This validator wasn't scored in the latest round. Validators appear in       │
│ rounds automatically once they're active on the network — no registration    │
│ required.                                                                    │
└──────────────────────────────────────────────────────────────────────────────┘
```

Copy is actionable, not scolding. The specific round number is not mentioned; readers who want to know which round it is click through via the footer link.

### Failed-latest-round fallback

If the latest attempted round has status `FAILED` and its round number is newer than the latest `COMPLETE` round that the panel is rendering data from, the panel header gains an inline link to the failed round's detail:

```
SCORING                                              round #N+1 failed — see why
                                                      ↑ link to /unl-scoring/rounds/N+1
```

The panel body continues to show the most recent `COMPLETE` round's data (ring + dimension rows). The link text is concrete; clicking opens the failed round's detail page so operators can see why it failed.

### Mobile

The two-half body collapses to a single column at narrow widths (<720 px): ring and badge on top, dimension rows beneath. No layout rewrite — a single `flex-direction` change on the body container.

### Graceful failure

When the scoring service is unreachable (proxy returns 502 with no warm cache) or the network has no completed rounds yet, `useScoringContext` yields `null` for its `context` field and the entire Scoring section is skipped — the validator detail page falls through to its pre-scoring layout with no broken UI, no empty placeholders, no retry loops.

## Explorer-Side Architecture: Proxy + Caching

The explorer **does not** call the scoring service directly from the browser. Instead, the explorer's Express server (`explorer/server/`) proxies scoring endpoints at `/api/scoring/*`. This matches the existing `/api/v1/*` pattern and avoids CORS entirely (browser only ever calls its own origin).

### Proxy shape

```js
app.get('/api/scoring/:path(*)', async (req, res) => {
  const upstream = `https://scoring-${env}.postfiat.org/api/scoring/${req.params.path}${qs}`
  // … caching layer, then fetch, then return …
})
```

### Caching (stale-while-revalidate, per-endpoint TTL)

The proxy maintains a process-wide in-memory cache keyed by URL (including query string). TTLs are per-endpoint:

| Endpoint | Why | Suggested TTL |
|---|---|---|
| `GET /rounds/<N>` (past round) | Immutable once round is complete | Indefinite (or 24h for safety) |
| `GET /rounds?limit=20` | Changes on each new round (~weekly) | A few minutes |
| `GET /unl/current` | Changes on each new successful round | A few minutes |
| `GET /rounds?limit=1` | May flip mid-round (running → complete) | 15–30s |
| `GET /config` | Rarely changes | 1 hour |

**Stale-while-revalidate**: on cache miss or expiry, try upstream. On success, update cache. On upstream failure, **serve the cached (stale) value** with a response header `X-Scoring-Stale: true` so the UI can show a subtle "showing cached data — scoring service unreachable" notice.

**Benefits:**

1. First-time visitors during transient scoring-service downtime still get data (cache is shared across all visitors, not per-browser).
2. Normal operation is sub-100ms (cache hits) instead of hitting the scoring service on every page load.
3. Load on the scoring service drops to ~1 upstream call per TTL window regardless of explorer traffic.
4. Historical rounds are effectively free — immutable, cached indefinitely.

**Cache eviction**: process restart resets cache (acceptable — in-memory, no persistence). No Redis or on-disk cache needed for Phase 1.

**Cold-start failure** (Express just booted + scoring service down + no cache): falls through to the graceful-degrade behavior below.

## Loading / Error / Empty States

| State | Behavior |
|---|---|
| **Loading** | Skeleton rows in tables, shimmer in badges, empty bars in dimensions. Never show `— no data` during load. |
| **Genesis** — no completed rounds ever on this network | Hide the Scoring nav link, hide the Status column on the Validators page, hide the Scoring section on the validator detail page. If someone hits `/unl-scoring` directly: show `No scoring rounds have completed on this network yet.` |
| **Transient error with cached data** | Serve cached data with `showing cached data — scoring service unreachable` banner. UI stays fully functional. |
| **Transient error, no cache** (cold start + scoring down) | Scoring page shows `Can't reach scoring service right now — try again in a moment.` Other pages degrade: Status column hidden with a small inline notice. |
| **Config endpoint failure** | Banner countdown shows `—`. Methodology section shows prose without live values (not hardcoded fallbacks). |
| **Failed round selected** | Audit panel hidden (see Section E). Round history row shows `Failed at <stage>`. Banner shows A.3 variant. |

Auto-detection of genesis vs active is based on `GET /rounds?limit=1`: empty result → genesis state, non-empty → feature is live. No env flag. When the first round completes on a new network, the UI appears automatically without coordinated redeploy.

## Accessibility & Mobile

### Accessibility

- Status states use **distinct glyphs** (`● ◐ ○ —`), not color alone. Glyphs render as actual distinct characters, not CSS-tinted dots.
- Color palette differences (green / amber / grey / light-grey) are secondary signaling. The glyph always conveys state.
- Interactive elements (table rows, accordion headers, sortable column headers) are keyboard-accessible with visible focus rings.
- Color contrast on bars and badges meets WCAG AA.

### Mobile

The 7-column ranked table doesn't fit on a phone. On narrow viewports:

- The 5 dimension columns collapse into a single `Details ▼` cell. Tapping the arrow expands the row to show the 5 dimensions inline below the overall score.
- Columns `Rank`, `Validator`, `Overall`, `Δ`, `Details ▼` remain visible.
- The drill-down (Section C) expands below the row as on desktop.

Other surfaces:

- Validators page table: existing mobile behavior, extended to the new Status column. Three Agreement columns may collapse per existing responsive rules.
- Validator detail Scoring section: renders as a single-column stack on narrow viewports.

Mobile layout must be verified on devnet before deploy, not left for polish.
