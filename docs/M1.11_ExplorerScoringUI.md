# Explorer Scoring UI — Design Sketch

Design proposal for the scoring surfaces introduced in M1.11 (Explorer Scoring Page). This document defines the information architecture and page layout for the new UI so the shape can be reviewed and agreed on before implementation begins. Visual styling, typography, and component-level design decisions are deferred to the frontend implementation pass.

Three pages are touched:

1. **Validators page** (existing) — gets a score badge column
2. **UNL Scoring page** (new) — dedicated page in the explorer navigation
3. **Validator detail page** (existing) — gets a compact scoring section with status, overall score, and dimension breakdown

---

## Page 1: Validators Page (Enhanced)

The existing validators table (`explorer/src/containers/Network/ValidatorsTable.tsx`) currently has the following columns: **Pubkey**, **Domain**, **UNL** (a binary green checkmark rendered when the validator is on the UNL), **Version**, the agreement scores **1H / 24H / 30D** (on the uptime tab), the fee-voting fields (on the voting tab), and **Last Ledger**.

The existing binary **UNL** column is **replaced** by two new columns: **Score** and **Status**. This is strictly richer than the current column — the Status badge still conveys UNL membership (a green `● on UNL` replacing the existing checkmark) and additionally expresses the `candidate` and `ineligible` states the current column cannot represent. The Score column surfaces the numeric overall score from the latest round and makes the table sortable by score.

Before (relevant slice):

| Pubkey | Domain | UNL | Version | ... |
|---|---|---|---|---|
| nHUKG1ZY… | postfiat.org | ✓ | 1.0.3 | ... |

After (relevant slice):

| Pubkey | Domain | Score | Status | Version | ... |
|---|---|---|---|---|---|
| nHUKG1ZY… | postfiat.org | 82 | ● on UNL | 1.0.3 | ... |

Both new columns are always visible (not tab-gated), matching the visibility of the UNL column they replace. All other existing columns (Version, Agreement 1H/24H/30D, Fee Voting fields, Last Ledger) are untouched.

```

│  Block Explorer                                  [ Ledgers  Validators  ▸] │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │

│                                                                            │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │ Public Key     Domain               Agreement   Score     Status     │  │
│  ├──────────────────────────────────────────────────────────────────────┤  │
│  │ nHUKG1ZY...    postfiat.org          99.8%    [  82  ]   ● on UNL    │  │
│  │ nHUiSBus...    postfiat.org          99.7%    [  79  ]   ● on UNL    │  │
│  │ nHUddEF9...    postfiat.org          99.9%    [  75  ]   ● on UNL    │  │
│  │ nHUxxxxxX...   acme-validators.io    98.1%    [  58  ]   ◐ candidate │  │
│  │ nHUyyyyyY...   (unverified)          82.4%    [  52  ]   ◐ candidate │  │
│  │ nHUvvvvvV...   (unverified)          79.1%    [  31  ]   ○ ineligible│  │
│  │ nHUzzzzzZ...   (unverified)           —       [  —   ]   — no data   │  │
│  │  ...                                                                 │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                            │
│  Scores from round #14 — completed 2 days ago.  [ View scoring details → ] │
└────────────────────────────────────────────────────────────────────────────┘
```

**Status values** derive directly from the UNL selection algorithm (`scoring_service/services/unl_selector.py`). The states are **mutually exclusive and exhaustive** for any validator in a given round:

| UI label | Exact condition | Data source |
|---|---|---|
| ● on UNL | `score ≥ cutoff` AND in the final selected UNL | `unl.json` → `primary` |
| ◐ candidate | `score ≥ cutoff` AND NOT in the final UNL | `unl.json` → `alternates` |
| ○ ineligible | `score < cutoff` | present in `scores.json`, absent from `unl.json` |
| — no data | validator not in this round's scoring evidence at all | absent from `scores.json` |

A validator is in exactly one category per round. A validator **cannot** be both `candidate` and `ineligible` — the selection algorithm only draws alternates from the pool of validators that already cleared the eligibility cutoff.

Note on churn: in rounds after the first, `alternates` can include validators that were previously on the UNL but got displaced by stronger challengers this round. They are still `candidate`s (above cutoff, not selected). The UI does not need to distinguish displaced incumbents from first-time challengers.

**Other notes:**
- Badge color: green (on UNL), amber (candidate), grey (ineligible), light grey (no data)
- The badge is display-only. Navigation to the Scoring page uses the `[ View scoring details → ]` link below the table.
- Clicking a validator row still goes to the existing validator detail page (unchanged behavior).

## Page 2: UNL Scoring (New Page)

Added to the top-level explorer navigation. Single-page layout with stacked sections.

### Section A — Header banner: round state + countdown

The banner's appearance depends on pipeline state. Three variants:

### Section B — Ranked validator table

The core view: all scored validators, sorted by overall score descending. The table includes **two separator lines** corresponding directly to the two configurable parameters of the selection algorithm:

1. **UNL selection line** — drawn between the last validator selected onto the UNL and the first candidate. Size of the UNL is set by the operator and will vary between environments.
2. **Cutoff line** — drawn between the last candidate and the first validator scoring below the eligibility cutoff.

Illustrative example (the actual number of rows per zone depends on the round and environment):

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  RANKED VALIDATORS — Round #14                                              │
│                                                                             │
│  Rank  Validator      Overall  Consensus Reliab. Software Diversity Identity│
│  ───────────────────────────────────────────────────────────────────────────│
│   1    nHUKG1ZY...      82      85       92       78       85       70      │
│   2    nHUiSBus...      79      82       90       75       80       68      │
│   3    nHUddEF9...      75      80       88       72       72       63      │
│         ...                                                                 │
│  ═════════════════════════ UNL selection line ══════════════════════════════│
│         ...                                                                 │
│   N+1  nHUxxxxX...      58      60       55       62       58       55      │
│   N+2  nHUyyyyY...      52      50       48       60       55       47      │
│  ─────────────────────────── Cutoff line ───────────────────────────────────│
│   N+3  nHUzzzzZ...      31      25       30       40       35       25      │
│                                                                             │
│  [ ▸ Click a row to see the LLM's reasoning for that validator ]            │
└─────────────────────────────────────────────────────────────────────────────┘
```

Rows above the selection line are `on UNL`. Rows between the two lines are `candidate`s — they would enter the UNL if a slot opened AND they cleared the churn gap against the weakest incumbent. Rows below the cutoff line are `ineligible`.

### Section F — Methodology explainer

Collapsed by default, expanded on click. First-time visitors can understand the system without leaving the page.

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  ▸ How does scoring work?                                                    │
│  ▸ What do the 5 dimensions mean?                                            │
│  ▸ How are validators chosen for the UNL?                                    │
│  ▸ How can I verify this is authentic?                                       │
└──────────────────────────────────────────────────────────────────────────────┘
```

Expanded "How are validators chosen for the UNL?" example:

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  ▼ How are validators chosen for the UNL?                                    │
│                                                                              │
│  Validators are scored 0–100 by an open-weight LLM across 5 dimensions.      │
│  A validator must score at or above the eligibility cutoff (currently        │
│  {cutoff}) to qualify. The top {max_size} qualifying validators are          │
│  placed on the UNL for the coming round.                                     │
│                                                                              │
│  To prevent churn, a challenger must score at least {min_gap} points         │
│  higher than the weakest incumbent to displace it.                           │
│                                                                              │
│  (Values in braces are fetched live from the scoring service config.)        │
│                                                                              │
│  For full methodology, see: docs/Design.md                                   │
└──────────────────────────────────────────────────────────────────────────────┘
```

The live values (cutoff, max size, min gap, cadence) are fetched from the new `/api/scoring/config` endpoint — never hardcoded in the frontend.

## Page 3: Validator Detail Page (Enhanced)

The existing validator detail page (per-validator deep link in the explorer) gains a new **Scoring** section. This is deliberately compact — the full ranked context, reasoning text, and audit trail live on the Scoring page and are reached via a single link.

### Standard case — validator was in the latest round

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  Scoring — Round #14 (2 days ago)                                            │
│                                                                              │
│  Status: ● on UNL               Overall: 82                                  │
│                                                                              │
│    Consensus   Reliability   Software   Diversity   Identity                 │
│       85           92           78          85          70                   │
│                                                                              │
│  [ View reasoning and round history → ]                                      │
└──────────────────────────────────────────────────────────────────────────────┘
```

The `[ View reasoning and round history → ]` link navigates to the Scoring page and scrolls to this validator's row with the drill-down auto-expanded.

### No-data case — validator not in the latest round

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  Scoring                                                                     │
│                                                                              │
│  Not scored in the latest round (#14). A validator must be seen by the       │
│  scoring pipeline to be included in a round.                                 │
│                                                                              │
│  [ Learn more → ]                                                            │
└──────────────────────────────────────────────────────────────────────────────┘
```

The `[ Learn more → ]` link opens the methodology section of the Scoring page.

**Design notes:**
- Reasoning text is intentionally omitted here. It belongs on the Scoring page where the full round context (network summary, cutoff lines, peer validators) is available alongside it.
- The five dimension scores are rendered inline (not in a chart) for compactness and to match the header row of the ranked table on the Scoring page.
- The status badge uses the same color rules as the Validators page: green / amber / grey / light grey for `on UNL` / `candidate` / `ineligible` / `no data`.
- If the latest round is `FAILED`, the Scoring section falls back to the most recent `COMPLETE` round and adds a note: *"Showing scores from round #N — round #N+1 failed (tap for details)."*

## Data Sources Per Section

| UI element | Endpoint |
|---|---|
| Score badge on Validators page | `GET /api/scoring/unl/current` + `GET /api/scoring/rounds?limit=1` |
| Validator detail page — Scoring section | `GET /api/scoring/rounds?limit=1` + `GET /api/scoring/unl/current` — resolve status and dimension scores by master key |
| Header banner — current round | `GET /api/scoring/rounds?limit=1` |
| Header banner — countdown | `GET /api/scoring/config` (cadence) + last `completed_at` |
| Ranked validator table | `GET /api/scoring/rounds/<N>` → `scores.json` |
| Per-validator drill-down | Same as above |
| Round history list | `GET /api/scoring/rounds?limit=20` |
| Audit trail panel | Round detail → `metadata.json` + memo tx hash |
| File download buttons | `GET /rounds/<N>/{snapshot,scores,unl,vl,metadata}.json` |
| IPFS gateway links | Built from `ipfs_cid` in round detail |
| On-chain memo link | Built from `memo_tx_hash` — links to block explorer |
| Methodology live values | `GET /api/scoring/config` |

## Backend Additions Needed

One new endpoint on the scoring service:

**`GET /api/scoring/config`** — read-only public config:

```json
{
  "cadence_hours": 168,
  "unl_score_cutoff": 40,
  "unl_max_size": 35,
  "unl_min_score_gap": 5
}
```

No other backend changes required. Everything else the UI needs is already exposed.
